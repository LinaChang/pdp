系統/平台開發流程說明

如何使用這份文件

這是公司系統/平台開發流程（PDP）的標準作業程序（SOP）。不論你來自行銷、業務、客服或其他單位，只要參與產品功能的開發，都會用到它。第一次閱讀的人，建議照下面的順序進行。

1
先看下方的常見誤區，建立對流程的正確心態。
2
遇到看不懂的英文縮寫（如 PRD、UAT），到上方的名詞對照表頁籤查詢。
3
用下方的流程總覽快速表建立整體輪廓，了解每個階段在做什麼。
4
依實際情境，深入閱讀自行開發流程或委外廠商流程頁籤。每個階段卡片都可點擊展開，看到參與者、產出文件與「AI 可以幫什麼」。
常見誤區
迷思 有了 AI，需求討論可以跳過，直接做出成品。
正解 AI 只是把想法執行得更快。需求沒講清楚，AI 只會更快做出錯的東西。需求確認與規格書，才是讓所有人對齊的基礎。
迷思 規格書（PRD）是工程師才需要看的技術文件。
正解 PRD 是所有人對齊的「合約」。行銷、業務、客服都應在它定案前，確認與自己相關的部分。
迷思 委外給廠商後，把需求丟過去、等成品交回來就好。
正解 委外只是換人執行。選商、合約、中期 Check-in、驗收一個都少不了，內部仍要全程把關。
迷思 流程跑得越快越好，文件能省則省。
正解 每個步驟都有存在的理由。跳過任何一步，都會在後面以返工、糾紛或上線出包的形式，用更高的成本補回來。
迷思 功能上線，專案就結束了。
正解 上線後要對照當初 MRD 設定的目標追蹤數據、收集回饋，才知道有沒有真正解決問題。

何時自行開發 vs 委外廠商
適合自行開發
  • 屬於核心功能，需要長期持續迭代與維護
  • 內部已具備對應的技術能力與人力
  • 需求預期會頻繁變動，需要快速調整
  • 涉及敏感資料或商業機密，不宜外流
適合委外廠商
  • 一次性、專案型或非核心的功能
  • 內部缺乏該領域的技術或人力
  • 規格明確且穩定，開發期間變動少
  • 時程緊迫，需要外部資源加速完成
關於這份文件

本頁說明團隊自行開發一個產品功能時，從概念到上線的完整流程。每個步驟都有其存在的理由——跳過任何一步，都會在後續付出更高的成本。AI 工具可以輔助加速部分工作，但無法取代需求確認與規格撰寫。

核心觀念

很多人以為有了 AI，就能「跳過」需求討論直接做出成品。但 AI 只能把你的想法執行得更快——如果需求本身沒說清楚，AI 只會更快地做出錯誤的東西。

規格書（PRD）不是寫一次就定稿，而是需要持續迭代的文件。從需求確認、開發、測試到落地上線，每一個環節都以它為準——規格在過程中每被釐清、修正一次，就少一次認知落差。它更是使用 AI 或其他開發工具時最關鍵的參考依據：你餵給工具的規格越清楚、迭代得越完整，產出才會越貼近真正想要的結果。規格沒迭代到位就開工，等於讓所有人（與 AI）各自猜，最後用返工的成本補回來。

名詞對照表

這份流程會用到不少英文縮寫與專有名詞。下面依主題分類，用白話解釋每個詞的意思。閱讀其他頁籤遇到不懂的詞，回到這裡查就對了。

需求與規格文件
MRD市場需求文件 · Market Requirements Document
說明「為什麼要做這個功能」、解決誰的問題，是需求發想階段的產出。別被「市場」二字誤導——它泛指「需求的來源與使用對象」；若是內部系統，這裡的「市場」指的就是內部使用單位，重點不在市場分析，而在確認「為誰解決什麼問題、值不值得做」。
PRD產品需求文件 · Product Requirements Document
定義功能「要做到什麼程度」的詳細規格，是開發的依據；委外時還會成為合約附件。
RFP需求徵求書 · Request for Proposal
對外發給廠商的文件，說明需求範圍，邀請廠商提出解決方案與提案。
RFQ報價徵求書 · Request for Quotation
與 RFP 類似，但聚焦在請廠商針對明確項目提出報價。
Definition of Done完成定義 · DoD
事先寫清楚「怎樣才算做完」的標準，避免驗收時雙方各說各話。
Launch Checklist上線清單
上線前逐項確認所有前置工作都已完成、沒有遺漏的檢查表。
Release Note版本說明
說明這次更新了哪些功能或修正的文件，可對內或對外發布。
設計與畫面
Wireframe線框圖
只畫出版面架構與元件位置的草圖，不含顏色與美術設計，用來討論結構。
Mockup視覺稿 / 高擬真設計稿
接近成品樣貌的設計稿，包含顏色、字體與圖像，讓人預覽完成後的畫面。
UI / UX使用者介面 / 使用者體驗
UI 指畫面的長相與元件；UX 指使用過程是否順暢、好懂、感受良好。

Alpha 版本
功能基本可用、僅供內部小範圍測試的早期版本，通常還不穩定。
Beta 版本
功能已完整，開放更廣範圍測試並開始收集使用者回饋的版本。
Release Candidate候選發行版本 · RC
已準備就緒，若測試沒問題就能正式上線的版本。
QA品質保證 · Quality Assurance
負責測試、確認功能正確性、效能與品質的角色或工作。
回歸測試Regression Testing
改動程式後重新測試原有功能，確認舊功能沒有被一起改壞。
UAT使用者驗收測試 · User Acceptance Testing
由需求方對照規格與合約，逐項確認交付物是否真的符合要求。
Edge case邊界情況 / 例外狀況
正常流程以外的特殊狀況（如空白輸入、極端數值），最容易被忽略。
專案與廠商管理
PM產品經理 · Product Manager
負責定義需求、協調各方、把關產出的角色，是整個流程的主導者。
Kick-off專案啟動會議
專案正式開始前，讓所有參與者對齊目標、分工與溝通方式的會議。
KT知識轉移 · Knowledge Transfer
廠商或前手向接手的人說明系統如何運作，確保後續能自行維護。
Retrospective回顧會議 · Retro
專案告一段落後，檢討「哪裡做得好、哪裡要改進」的會議。
評分矩陣Scoring Matrix
用多個維度（技術、經驗、報價等）為廠商打分、方便客觀比較的表格。
NPS淨推薦值 · Net Promoter Score
衡量使用者願不願意把產品推薦給別人的滿意度指標。
